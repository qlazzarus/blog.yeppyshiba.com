# 블로그 초안 — GA4 조회수 표시의 한계와 Cloudflare 자체 집계로 바꾼 이유

> 게시 전 상태: 구현 전 초안. Cloudflare Worker/D1 배포와 실제 동작 검증이 끝난 뒤에만 본문의 과거형 표현을 유지한다. 구현 범위가 달라지면 제목·구성·보존 기간을 함께 수정한다.

## 제목 후보

1. GA4 조회수를 사이트에 그대로 보여주지 않기로 한 이유
2. 정적 블로그에 자체 조회수와 좋아요를 붙인 방법
3. GitHub Pages 블로그에 Cloudflare Worker와 D1을 붙여 본 기록

## 메타 설명 후보

GA4는 훌륭한 분석 도구지만, 정적 블로그의 공개 조회수 표시에는 빈틈이 있었다. GitHub Pages는 유지하면서 Cloudflare Worker와 D1으로 자체 조회수·좋아요를 설계한 이유와 기준을 정리한다.

## 본문 초안

블로그 목록에 보이는 조회수는 꽤 단순한 숫자처럼 보인다. 하지만 그 숫자를 어디서 가져오고, 무엇을 하나의 조회로 볼지 결정하는 순간부터 생각보다 많은 선택이 생긴다.

이 블로그는 한동안 GA4 Data API에서 `screenPageViews`를 읽어 GitHub Actions 빌드 과정에 넣는 방식을 사용했다. GA4는 검색 유입, 방문 경로, 체류시간처럼 콘텐츠를 개선하는 데 필요한 분석에는 여전히 좋은 도구다.

그런데 화면에 보여 줄 “이 글은 얼마나 읽혔나”라는 숫자로는 몇 가지 아쉬움이 있었다.

## GA4의 숫자가 틀린 것은 아니었다

먼저 분명히 할 점이 있다. GA4가 반환한 숫자가 곧바로 오류라는 뜻은 아니다. GA4는 설정된 속성으로 수집된 페이지뷰를 충실하게 보여 준다.

문제는 그 숫자의 의미가 사이트에서 기대하는 공개 조회수와 정확히 같지 않다는 데 있었다.

- GA4 설치 이전의 방문은 집계에 없다.
- 광고 차단이나 추적 거부, JavaScript를 실행하지 않는 환경에서는 이벤트가 전송되지 않을 수 있다.
- 이전 URL과 현재 URL이 나뉘면 같은 글의 수치가 갈라질 수 있다.
- 정적 사이트는 새로 빌드·배포하기 전까지 화면의 숫자를 갱신할 수 없다.
- 분석용 속성·데이터 스트림 설정이 바뀌면 공개 숫자의 연속성을 관리하기 어렵다.

예를 들어 `/article/글-이름`과 `/article/글-이름/`은 사람이 보기에는 같은 페이지다. 집계 코드에서 이 차이를 합칠 수는 있지만, URL 변경 이력 전체를 계속 관리하는 일은 분석 도구에만 맡기기 어려웠다.

그래서 GA4는 **분석용**, 사이트 안의 조회수와 반응은 **서비스 기능용**으로 역할을 나누기로 했다.

## 정적 블로그는 그대로 두고, 작은 백엔드만 더했다

블로그 본문과 배포는 계속 GitHub Pages와 Astro를 사용한다. 글을 쓰고 정적 사이트를 배포하는 흐름은 이미 충분히 단순하고 안정적이기 때문이다.

대신 조회수와 좋아요처럼 새로고침 없이 변하는 기능만 Cloudflare에 분리했다.

```text
GitHub Pages / Astro
  ├─ 글을 열면 조회 이벤트 전송
  ├─ 좋아요를 누르면 반응 요청 전송
  └─ 글별 통계 조회
                 │
                 ▼
Cloudflare Worker
  ├─ 요청 검증과 중복 제한
  └─ Cloudflare D1에 집계 저장
```

이 구조를 택한 이유는 조회수 기능에 거대한 서버나 항상 켜져 있는 데이터베이스가 필요하지 않기 때문이다. Worker는 요청이 있을 때만 실행되고, D1은 이 프로젝트에 필요한 단순 SQL 집계에 잘 맞는다.

무엇보다 이 기능을 위해 블로그 전체를 다른 호스팅으로 옮길 필요가 없었다.

## 무엇을 한 번의 조회로 볼까

조회수는 새로고침 횟수가 아니다. 그렇게 세면 숫자는 쉽게 부풀고, 글이 실제로 읽혔는지 알기 어려워진다.

이 블로그에서는 다음 기준을 사용한다.

- 글이 화면에 실제로 보인 뒤 짧은 시간을 지난 경우에만 조회 이벤트를 보낸다.
- 같은 브라우저가 같은 글을 같은 날 다시 열어도 조회수는 한 번만 증가한다.
- 경로는 trailing slash, 쿼리스트링, 해시를 제거한 canonical path로 정규화한다.
- 좋아요는 같은 브라우저에서 글 하나당 하나만 남길 수 있고, 다시 누르면 취소된다.

이 기준은 “완벽한 순 방문자 수”를 만들기 위한 것은 아니다. 로그인도 없이 개인을 정확하게 구분하는 것은 가능하지도, 이 블로그에 필요한 일도 아니다. 대신 반복 새로고침이나 단순 자동 요청으로 숫자가 쉽게 흔들리지 않도록 하는 기준이다.

## 익명성은 어떻게 다뤘나

좋아요를 한 번만 누르게 하려면, 서버는 적어도 같은 브라우저가 다시 요청한 것인지는 알아야 한다.

처음에는 브라우저의 localStorage에 UUID를 저장하는 방법도 생각했다. 구현은 쉽지만 JavaScript에서 값이 보이고 쉽게 바꿀 수 있으며, 좋아요 상태를 믿기 어렵다.

대신 API가 불투명한 랜덤 값을 담은 HttpOnly 쿠키를 발급하도록 했다. JavaScript는 이 값을 읽을 수 없고, 요청할 때만 브라우저가 API로 보낸다.

데이터베이스에는 쿠키의 원본 값을 저장하지 않는다. 서버만 아는 비밀 값으로 만든 HMAC 해시만 저장한다.

- 조회수는 `익명 쿠키 + 날짜`를 해시해 같은 날의 중복만 막는다.
- 좋아요는 `익명 쿠키`를 해시해 같은 글의 중복을 막고, 취소 여부를 판단한다.
- IP 주소, 이메일, 브라우저 지문, 상세 referrer는 저장하지 않는다.

쿠키를 지우거나 다른 기기에서 접속하면 새 익명 사용자로 보인다. 이것은 의도한 한계다. 더 강한 식별을 위해 개인 데이터를 늘리는 것보다, 가벼운 반응 기능에는 이 정도의 익명성이 더 적절하다고 판단했다.

## GA4는 계속 남긴다

자체 조회수가 생겼다고 GA4를 없애지는 않는다.

GA4는 어떤 검색어와 채널에서 방문했는지, 어떤 글에서 다음 글로 이동했는지, 어느 콘텐츠의 체류시간이 긴지를 보는 분석 도구다. 자체 집계는 사이트에 표시할 조회수와 좋아요를 안정적으로 운영하는 기능이다.

둘의 숫자가 항상 같을 필요도 없다. 수집 기준이 다르기 때문이다. 오히려 차이를 통해 추적 차단이나 URL 분산, 특정 콘텐츠의 반응을 더 잘 이해할 수 있다.

## 다음 단계

처음에는 조회수와 좋아요만 다룬다. 충분히 안정적으로 운영되면 다음 같은 기능도 같은 작은 백엔드 위에 얹을 수 있다.

- “도움됐어요” 같은 가벼운 반응
- 오탈자·링크 오류 제보
- 글별 반응 추이

하지만 시작은 작게 하려고 한다. 공개 숫자는 정확한 척하는 것보다, 무엇을 세고 무엇을 세지 않는지 분명한 편이 더 낫다고 믿기 때문이다.

## 게시 전 체크리스트

- [ ] Worker와 D1이 실제 운영 환경에서 배포·검증되었다.
- [ ] 쿠키 이름, 보존 기간, API 도메인을 실제 값으로 반영했다.
- [ ] 조회수·좋아요의 UI 라벨과 오류 fallback을 확인했다.
- [ ] 개인정보처리방침에 익명 쿠키 기반 집계를 반영했다.
- [ ] 실제 구현이 본문과 일치하는지, 특히 "IP를 저장하지 않는다"는 표현을 재검증했다.
- [ ] Cloudflare와 GA4의 숫자 차이를 두고 과장된 정확성 주장을 하지 않았는지 검토했다.
