# 닭농장 1.3A W3X 추출 분석 노트

> 대상 파일: `docs/닭농장1.3a.w3x`  
> 목적: 원본 맵의 에셋/코드 재사용이 아니라, Phaser 기반 오리지널 게임의 밸런스 기준과 플레이 루프 참고용 수치 정리

---

## 1. 추출 범위와 원칙

이 문서는 `닭농장1.3a.w3x`에서 추출 가능한 메타데이터, 문자열, JASS 트리거, 오브젝트 데이터의 관찰 결과만 정리한다.

원본 JASS 코드, 원본 에셋, 원본 모델/아이콘/사운드는 프로젝트에 복사하지 않는다. 구현에는 아래 수치와 구조를 참고하되, 이름, 그래픽, UI, 사운드, 코드 구조는 새로 만든다.

---

## 2. MPQ 추출 상태

맵 파일은 일반 zip/tar로는 열리지 않는다. 파일 내부에는 MPQ 헤더가 있으나, 헤더 일부에 `WOLF` 값이 들어가 있어 일반 MPQ 도구가 실패할 수 있는 보호/변조 흔적이 있다.

그래도 표준 MPQ 해시 기반 조회로 다음 파일은 추출 가능했다.

| 내부 파일            |     추출 크기 | 분석 용도                               |
| -------------------- | ------------: | --------------------------------------- |
| `war3map.wts`        |   1,527 bytes | 맵 이름, 설명, 제작자, 안내 문구        |
| `war3map.w3i`        |     784 bytes | 맵 정보, 플레이어/팀 메타데이터         |
| `scripts\war3map.j`  | 311,745 bytes | 트리거, 타이머, 자원 루프, 웨이브, 점수 |
| `war3map.w3e`        |  53,040 bytes | 지형 데이터                             |
| `war3map.wpm`        | 118,288 bytes | 경로 데이터                             |
| `war3map.doo`        |  58,124 bytes | 배치 오브젝트                           |
| `war3map.w3a`        | 227,666 bytes | 커스텀 능력 데이터                      |
| `war3map.w3b`        |     235 bytes | 커스텀 버프 데이터 일부                 |

| `war3map.w3h`        |  35,326 bytes | 커스텀 효과/기타 오브젝트 데이터        |

| 내부 파일                     | 상태                | 영향                                               |

| `war3map.w3u`                 | 표준 이름 조회 실패 | 익명 MPQ INI/SLK 블록에서 유닛 스탯 교차확인 완료  |

## 4. 플레이어/팀 구조

| 항목        | 관찰                           |
| ----------- | ------------------------------ |

| 유저 슬롯   | 1~10번 플레이어                |
| 컴퓨터 슬롯 | 11~12번 플레이어               |

Phaser MVP 반영:

- 원본은 최대 10인 구조지만 웹 MVP는 2~4인을 우선한다.
- 원본의 다인 경쟁 구조는 "각자 농장 + 공통 늑대 압박" 감각으로 축소한다.

## 7. 닭 경제 루프

JASS 후반부에 30초 주기의 자원 지급 트리거가 있다.

| 항목              |     값 |
| ----------------- | -----: |
| 지급 주기         |   30초 |
| 기본 닭 rawcode   | `h00A` |
| 기본 닭 목재 수입 |     70 |
| 중간 닭 rawcode   | `h00J` |
| 중간 닭 목재 수입 |    110 |
| 고급 닭 rawcode   | `h00W` |
| 고급 닭 목재 수입 |    170 |

원본은 닭 수에 따라 목재가 증가한다. 웹 게임에서는 자원명을 그대로 따르지 않고, 다음 중 하나로 해석한다.

| 원본       | 웹 MVP 후보                       |
| ---------- | --------------------------------- |
| 목재 지급  | 알 판매 수익 또는 농장 생산 수익  |
| 30초 주기  | 알 생산/수확 주기                 |
| 70/110/170 | 닭 등급별 생산량 또는 판매가 배율 |

권장 MVP 변환:

```json
{
    "chickenTiers": [
        { "tier": 1, "sourceRawcode": "h00A", "referenceYield": 70, "mvpEggAmount": 1 },
        {
            "tier": 2,
            "sourceRawcode": "h00J",
            "referenceYield": 110,
            "mvpEggAmount": 2
        },
        { "tier": 3, "sourceRawcode": "h00W", "referenceYield": 170, "mvpEggAmount": 3 }
    ],
    "eggProductionIntervalSec": 30
}
```

### 8.1 늑대 명령 좌표와 대상 우선순위

JASS 늑대 명령 흐름은 `jass_wolf_order_flows.tsv`, `jass_wolf_spawn_points.tsv`, `jass_wolf_order_areas.tsv`로 분리했다.

핵심 결론:

- JASS에는 "수익 건물 -> 닭 -> 플레이어 -> 펜스" 같은 구체적인 대상 우선순위가 직접 하드코딩되어 있지 않다.
- 일반 늑대는 13개 외곽 스폰 rect에서 등장하고, `IlIlI` 전체 공격 구역 안의 랜덤 지점으로 `attack` 명령을 받는다.
- 60초마다 Player(10)의 늑대 전체에 같은 방식의 전역 공격 갱신이 들어간다.
- 실제 공격 대상 선택은 Warcraft III의 attack-move, 자동 target acquisition, pathing, 충돌/막힘 처리에 맡겨진 구조로 보는 편이 맞다.
- 특수 이벤트성 명령은 보스/특수 어그로, 저체력 방어 유닛 집중, 수동 탱커 집중에 가깝고 일반 늑대의 기본 우선순위로 보기는 어렵다.

주요 좌표:

| 영역                  | 원본 변수    | 좌표                              |
| --------------------- | ------------ | --------------------------------- |
| 늑대 전역 공격 구역   | `IlIlI`      | `(-3168, -7424)` ~ `(7008, 2848)` |
| 난이도/특수 생성 구역 | `IliiI`      | 중심 `(2080, -1440)`              |
| 보스 생성 구역        | `IlilI`      | 중심 `(2080, -896)`               |
| 외곽 스폰 지점        | `Iii[1..13]` | `jass_wolf_spawn_points.tsv` 참조 |

확인된 명령 플로우:

| 함수                 |      라인 | 해석                                                                   |
| -------------------- | --------: | ---------------------------------------------------------------------- |
| `IliiIII`            | 5720-5722 | 늑대 스폰 rect에 들어온 Player(10) 유닛을 `IlIlI` 랜덤 지점으로 attack |
| `IliiIlI -> IliiIiI` | 5726-5728 | 60초마다 Player(10) 전체 유닛에 `IlIlI` 랜덤 지점 attack 갱신          |
| `llIIIi -> lillli`   | 3505-3507 | 트리거 유닛 주변 6000 범위 Player(10) 유닛을 해당 위치로 집중          |
| `lIlIIiI -> lIlIIII` | 9419-9430 | 저체력 `n00E` 주변 800 범위 적에게 해당 유닛 공격 명령                 |

Phaser MVP 해석:

- 전략 목표는 "농장 내부 가치 구역"으로 두되, 세부 우선순위는 Warsmash behavior 관찰을 바탕으로 웹 AI 레이어에서 새로 정의한다.
- 펜스/벽은 최우선 공격 대상이 아니라, 경로를 막거나 가까운 blocker가 되었을 때 공격하는 fallback 대상으로 둔다.
- 원본의 60초 전역 공격 갱신은 MVP에서는 idle/stuck 감지 후 3~8초 안에서 목표를 재평가하는 규칙으로 압축한다.

권장 MVP 변환:

| 원본 감각          | 웹 MVP 변환             |
| ------------------ | ----------------------- |
| 80초 경고          | 60~80초 첫 웨이브       |
| 600초 블러드 울프  | 6~8분 첫 엘리트         |
| 1500초 와일드 울프 | 10~12분 후반 압박       |
| 장기 보스          | MVP 이후 무한/하드 모드 |

## 12. 오브젝트 데이터 추가 분석

`war3map.w3a`, `war3map.w3q`, `war3map.w3d`는 raw object modification 형식으로 파싱 가능했다.

| 파일          | original 엔트리 | custom 엔트리 | 분석 의미         |
| ------------- | --------------: | ------------: | ----------------- |
| `war3map.w3a` |             314 |           170 | 커스텀 능력/스킬  |
| `war3map.w3q` |              56 |            15 | 커스텀 업그레이드 |
| `war3map.w3d` |               1 |             1 | 커스텀 장식물     |

`war3map.w3h`, `war3map.w3b`는 거의 끝까지 파싱되지만 마지막 바이트 정렬 문제로 완전 파싱은 실패했다. 현재 분석에는 `strings`/JASS 보조 근거만 사용한다.

### 12.1 알/부화/수집 능력

- 알은 바닥에 떨어지는 오브젝트로 만들 수 있지만, 초반 UX를 위해 자동 수집 반경을 기본 제공한다.
- "알 끌어모으기"는 추후 액티브 스킬 또는 건물 업그레이드로 재해석한다.
- 부화는 건물별 복잡도를 줄이고, MVP에서는 공통 `hatchery` 한 종류로 시작한다.

- 첫 버전에서는 모이 시스템을 별도 자원으로 만들지 않고, 닭 생산 가속 버프 정도로 제한한다.
- "닭 보호"는 늑대 웨이브 직전 쓰는 방어 스킬로 좋지만, MVP에서는 우물/울타리/타워가 먼저다.
- 특수 닭은 2차 버전의 전략 다양화 요소로 남긴다.

### 12.3 구매/교환/질병

JASS와 오브젝트 문자열에서 확인된 경제/상태 기능:

| 항목             | 원본 단서                                                          |
| ---------------- | ------------------------------------------------------------------ |
| 금/목재 교환     | 100 -> 70, 500 -> 400, 1500 -> 1200, 3000 -> 2400 비율의 상호 교환 |
| 중년 닭 구매     | `400` 골드로 중년 닭 1마리 구매                                    |
| 자이언트 닭 구매 | `850` 골드로 자이언트 닭 1마리 구매                                |
| 질병 치료        | `A02K`, `A032`, `A02O`                                             |
| 질병/질병 워드   | `A02L`, `A02M`                                                     |

MVP 반영:

- 금/목재 이중 경제는 웹 MVP에서 과할 수 있으므로 골드/알 중심으로 시작한다.
- 중년/자이언트 닭 구매가는 원본 비율만 참고한다.
- 질병은 MVP에서 제외하고, 추후 랜덤 재난/상태이상으로 검토한다.

## 16. JASS 함수군 라벨링

난독화된 JASS 함수명만으로는 역할을 바로 알기 어렵기 때문에, 호출 그래프에 기능군 라벨을 붙였다. 라벨은 함수 본문 키워드와 API 사용 패턴을 기준으로 1차 부여하고, 직접 라벨링된 함수를 호출하는 상위 함수에 한 단계 상속했다.

산출물:

| 파일                          | 내용                                                       |
| ----------------------------- | ---------------------------------------------------------- |
| `jass_function_labels.tsv`    | 함수별 라벨, 줄 범위, 직접/상속 라벨, 호출 함수, 라벨 근거 |
| `jass_label_summary.tsv`      | 라벨별 함수 수 요약                                        |
| `jass_labeled_call_edges.tsv` | 라벨링된 함수와 주변 호출 edge                             |

### 16.1 라벨별 세부 플로우

아래 플로우는 `jass_function_labels.tsv`, `jass_labeled_call_edges.tsv`, `jass_timer_events.tsv`, `jass_economy_events.tsv`, `jass_create_units.tsv`를 함께 대조한 결과다. 함수명은 난독화되어 있으므로 구현에서는 함수명을 복사하지 말고 역할 단위만 참고한다.

#### 웨이브 플로우

원본 웨이브는 `main`/`config`의 기본 세팅, `iiiIilI`의 트리거 등록, 5600~6800 라인대의 웨이브 단계 함수군으로 나뉜다.

```text
config/main
  -> 플레이어/팀/시작 위치/환경 초기화
  -> iiiIilI: 런타임 트리거와 장기 타이머 등록
  -> IliIilI: 난이도 선택 후 초기 늑대 계열 생성
  -> IliiiiI/IliilII/IliillI/IlilIiI/IliliII: 단계별 특수 웨이브
  -> IlillII~IllllII: 현재 단계 유닛 수가 줄면 Player(10)에 2기씩 보충
  -> iIIIilI~iIlIliI: 웨이브 카운트/상태 증가 계열
```

| 단계        | 근거 함수/라인                                       | 관찰                                                                                    |
| ----------- | ---------------------------------------------------- | --------------------------------------------------------------------------------------- |
| 초기화      | `config` 9685-9706, `main` 9670-9684                 | 시작 위치, 팀, 환경, 주요 초기화 함수 호출                                              |
| 트리거 등록 | `iiiIilI` 7233-7887                                  | `TriggerRegisterTimerEventSingle/Periodic`, 채팅/다이얼로그/유닛 이벤트를 한곳에서 등록 |
| 초기 웨이브 | `IliIilI` 5614-5663                                  | 난이도 선택 이후 Player(10) 소유 늑대 계열 생성과 안내                                  |
| 특수 웨이브 | `IliiiiI` 5741-5763                                  | Blood Wolf 계열로 보이는 단계. 생성, 핑, 시야 공유, 다음 트리거 활성화                  |
| 후속 단계   | `IliilII`, `IliillI`, `IlilIiI`, `IliliII` 5776-5894 | Wild Wolf, Hell Hound, Doom Guard, Archimonde 계열 단계로 추정                          |
| 보충 스폰   | `IlillII`~`IllllII` 5907-6174                        | 현재 단계 유닛 수 조건을 보고 `CreateNUnitsAtLoc(2, iIi[...], Player(10))` 형태로 보충  |
| 점수/상태   | `iiIIliI` 6943-6985                                  | 늑대 관련 안내와 리더보드 값 갱신                                                       |

타이머 등록상 장기 웨이브/이벤트 후보는 60, 120, 230, 300, 600, 1100, 1500, 2000, 2200, 2400, 2600, 2800, 3000초에 분포한다. Phaser MVP에서는 이 전체 길이를 그대로 쓰기보다 12~15분 세션에 맞춰 단계 간격을 압축하는 편이 좋다.

#### 경제 플로우

경제는 교환, 구매, 사망/부활 패널티, 주기 수익으로 분리된다.

```text
상점/유닛 상호작용
  -> lIilii: 골드/목재 교환
  -> lIlIii/lIliii/lIllii: 금화 지불 후 닭 계열 생성
  -> liilIi: 영웅 부활 및 자원 40% 손실
  -> lIiillI: 30초마다 닭 보유 수 기준 목재 지급
```

| 기능        | 근거 함수/라인      | 원본 수치/행동                                                               |
| ----------- | ------------------- | ---------------------------------------------------------------------------- |
| 자원 교환   | `lIilii` 3182-3255  | 100 -> 70, 500 -> 400, 1500 -> 1200, 3000 -> 2400 비율로 골드/목재 상호 교환 |
| 닭 구매 1   | `lIlIii` 3268-3275  | 금화 150 소모, Player(11) 소유 유닛 생성                                     |
| 닭 구매 2   | `lIliii` 3288-3295  | 금화 400 소모, Player(11) 소유 유닛 생성                                     |
| 닭 구매 3   | `lIllii` 3308-3315  | 금화 850 소모, Player(11) 소유 유닛 생성                                     |
| 부활 패널티 | `liilIi` 3415-3445  | 영웅 부활 처리, 골드/목재 현재 보유량의 40% 차감                             |
| 주기 수익   | `lIiillI` 9352-9366 | 30초마다 `h00A`/`h00J`/`h00W` 보유 수에 따라 목재 70/110/170 지급            |

Phaser에서는 교환 UI와 닭 구매 UI를 분리하고, 주기 수익은 서버/호스트 authoritative tick으로 처리하는 것이 좋다. 원본의 부활 패널티 40%는 협동 게임에서 긴장감을 만드는 장치라 MVP에도 유지할 만하다.

#### 질병 플로우

질병은 필수 웨이브라기보다 선택형 압박 모드에 가깝다. 다이얼로그/안내/트리거 활성화 후 일정 시간 생존하는 질병 유닛을 생성한다.

```text
질병 옵션 노출
  -> lliIIi: 질병 다이얼로그 버튼 표시
  -> lliiii: 선택 결과/안내 출력
  -> llliii: 질병 트리거 활성화와 안내
  -> llllIi/llllli: 질병 유닛 생성 후 timed life 적용
```

| 단계      | 근거 함수/라인               | 관찰                                                              |
| --------- | ---------------------------- | ----------------------------------------------------------------- |
| 선택 UI   | `lliIIi` 3547-3556           | `DialogAddButtonBJ`, `DialogDisplayBJ`, `DialogSetMessageBJ` 사용 |
| 안내/분기 | `lliiii` 3575-3592           | 질병 관련 텍스트 출력과 조건부 트리거 실행                        |
| 모드 시작 | `llliii` 3642-3660           | 질병 안내, 트리거 활성화, 시간값 `Ili[60]` 참조                   |
| 효과 생성 | `llllIi`, `llllli` 3667-3686 | Player(10) 또는 중립 적대 유닛 생성, `UnitApplyTimedLife` 적용    |

웹 버전에서는 질병을 별도 룰셋으로 두고, 기본 모드에서는 꺼두는 것이 안전하다. 켠다면 “일정 시간마다 농장 내부에 오염/감염 이벤트 발생”처럼 원본의 timed life 압박만 가져오면 된다.

#### 채팅/명령 플로우

채팅 명령은 두 층이다. `iiiIilI`가 기본 런타임 이벤트를 등록하고, `lIlIiII`가 다수의 채팅 명령 등록 함수를 묶어서 호출한다.

```text
main
  -> lIlIiII: 채팅 명령 등록 묶음
       -> iiiIllI: 동맹 관련 명령
       -> iiiiiII: -전체
       -> iiiiiiI/iiiiilI: -모두
       -> iiiillI/iiilIlI: 조건부 채팅 명령
       -> illlIII: -이름
       -> lIIliiI: 추가 채팅 명령
  -> iiiIilI: 런타임 트리거 등록 허브
```

| 명령군      | 근거 함수/라인                           | 관찰                                               |
| ----------- | ---------------------------------------- | -------------------------------------------------- |
| 등록 허브   | `lIlIiII` 9437-9491                      | 53개 호출을 가진 채팅/동맹 등록 묶음               |
| 전체 채팅   | `iiiiiII` 7929-7942                      | `-전체` 채팅 이벤트 등록                           |
| 모두 채팅   | `iiiiilI` 7947-7960                      | `-모두` 채팅 이벤트 등록                           |
| 동맹 채팅   | `iiiIllI` 7895-7908                      | `-동맹` 관련 채팅 이벤트 등록                      |
| 이름 변경   | `illlIII` 8981-8994, `illillI` 8978-8980 | `-이름` 명령 등록과 `SetPlayerName` 실행           |
| 카메라 조정 | `IIllilI`, `IIlllII` 4273-4285           | 채팅 문자열 파싱 후 `SetCameraFieldForPlayer` 호출 |

Phaser에서는 원본 채팅 명령을 그대로 노출할 필요는 낮다. 개발자용 명령, 이름 변경, 전체/팀 채팅 정도만 현대 UI로 옮기면 충분하다.

#### 동맹 플로우

동맹은 채팅 명령과 다이얼로그가 함께 존재한다. 즉시 전체 동맹을 설정하는 함수와, 다이얼로그에서 개별 플레이어를 고르는 함수군이 분리되어 있다.

```text
채팅/동맹 명령
  -> IiIIilI: 전체 동맹/동맹 상태 계열 처리
  -> IiilliI: 동맹 대상 선택 다이얼로그 구성
  -> IilIIlI~IlIiIlI: 플레이어별 선택 액션
  -> IlIilII: 실제 SetPlayerAllianceStateBJ 적용 helper
```

| 단계            | 근거 함수/라인                | 관찰                                                                             |
| --------------- | ----------------------------- | -------------------------------------------------------------------------------- |
| 전체 동맹       | `IiIIilI` 4323-4350           | `SetPlayerAllianceStateBJ` 호출과 동맹 안내 출력                                 |
| 다이얼로그 구성 | `IiilliI` 4669-4946           | `DialogAddButtonBJ`, `DialogDisplayBJ`, `DialogSetMessageBJ`로 대상 선택 UI 구성 |
| 개별 대상 처리  | `IilIIlI`~`IlIiIlI` 4965-5426 | 버튼 선택 후 `DialogClearBJ`, 안내 출력, 동맹 상태 변경                          |
| 적용 helper     | `IlIilII` 5451-5458           | `SetPlayerAllianceStateBJ` 래퍼성 helper                                         |

웹 MVP에서는 워크래프트식 임의 동맹보다 방 생성 시 팀 고정/협동 고정이 더 단순하다. 단, 원본의 `-동맹`, `-올동맹`, `-동맹상태` 흐름은 P2P 방에서 “팀 상태 보기”와 “협동 초대/수락” UI 참고로 쓸 수 있다.

- Stratagus는 GPL-2.0이라 코드 재사용보다 구조 연구용으로만 둔다.

### 18.2 늑대 이동/공격 감각 재현 포인트

원본에서 확인된 사실:

| 근거                          | 관찰                                                                             |
| ----------------------------- | -------------------------------------------------------------------------------- |
| `war3map.wpm`                 | 기본 지형은 ground open 95.6%로 거의 열린 평면                                   |
| `jass_create_units.tsv`       | 늑대/보스는 대부분 Player(10)에 생성                                             |
| `jass_labeled_call_edges.tsv` | `IssuePointOrderLocBJ`, `IssueImmediateOrderBJ` 계열 명령 존재                   |
| `unit_rawcode_crosscheck.tsv` | 일반 늑대 속도 330, 보스 대부분 속도 330                                         |
| `war3map.doo`                 | 농장 주변 doodad는 장식/경계 감각은 주지만 기본 pathing을 미로처럼 만들지는 않음 |

재현 방향:

1. 늑대는 타워디펜스의 고정 경로 적이 아니라, 열린 지형에서 농장/닭/수익 건물을 향해 들어오는 추적자에 가깝게 둔다.
2. 기본 목표 우선순위는 `수익 건물 -> 닭/방어 유닛 -> 농부/핵심 건물`로 둔다.
3. 펜스가 없으면 거의 직선 추적을 하며, 펜스나 건물로 막히면 가까운 우회로를 찾는다.
4. 우회 실패가 길어지면 워3식으로 근처 blocker를 공격 대상으로 전환한다.
5. 공격은 사거리/쿨다운 기반이며, 타깃이 죽거나 사라지면 즉시 재탐색한다.

### 18.3 펜스/장애물 감각 재현 포인트

원본에 명시적인 “펜스” 이름으로 추출된 유닛 스탯은 아직 별도 확정되지 않았다. 다만 `war3mapImported\GreenGrass_Wall.mdx` 참조 에셋과 doodad/pathing 결과를 보면, 플레이어가 방어선을 직접 만들고 늑대의 접근을 지연시키는 감각이 핵심이다.

Phaser에서의 펜스 모델은 다음을 목표로 한다.

| 속성           | 권장                                 |
| -------------- | ------------------------------------ |
| pathing        | grid cell blocker                    |
| 공격 대상 여부 | 늑대가 목표로 삼을 수 있음           |
| 내구도         | 초반 늑대 3~5회 공격에 파괴되는 수준 |
| 배치 규칙      | 플레이어 농장 구역 내부만 가능       |
| 수리/교체      | MVP에서는 재건설 우선, 수리는 후순위 |
| 시각 감각      | 낮고 연속된 울타리/생울타리 실루엣   |

핵심은 “완벽한 미로 만들기”가 아니라 “잠깐 늑대를 묶어두는 방어선”이다. 원본 pathing이 넓게 열려 있으므로, 웹 버전에서도 펜스는 긴 우회 미로보다 농장 보호용 지연 장치로 두는 편이 원본 감각에 가깝다.
