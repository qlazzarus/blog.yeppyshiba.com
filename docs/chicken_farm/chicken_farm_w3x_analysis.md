# 닭농장 1.3A W3X 추출 분석 노트

> 대상 파일: `docs/닭농장1.3a.w3x`  
> 목적: 원본 맵의 에셋/코드 재사용이 아니라, Phaser 기반 오리지널 게임의 밸런스 기준과 플레이 루프 참고용 수치 정리

---

## 1. 추출 범위와 원칙

이 문서는 `닭농장1.3a.w3x`에서 추출 가능한 메타데이터, 문자열, JASS 트리거, 오브젝트 데이터의 관찰 결과만 정리한다.

원본 JASS 코드, 원본 에셋, 원본 모델/아이콘/사운드는 프로젝트에 복사하지 않는다. 구현에는 아래 수치와 구조를 참고하되, 이름, 그래픽, UI, 사운드, 코드 구조는 새로 만든다.

---

## 2. MPQ 추출 상태

맵 파일은 일반 zip/tar로는 열리지 않는다. 파일 내부에는 MPQ 헤더가 있으나, 헤더 일부에 `WOLF` 값이 들어가 있어 일반 MPQ 도구가 실패할 수 있는 보호/변조 흔적이 있다.

그래도 표준 MPQ 해시 기반 조회로 다음 파일은 추출 가능했다.

| 내부 파일            |     추출 크기 | 분석 용도                               |
| -------------------- | ------------: | --------------------------------------- |
| `war3map.wts`        |   1,527 bytes | 맵 이름, 설명, 제작자, 안내 문구        |
| `war3map.w3i`        |     784 bytes | 맵 정보, 플레이어/팀 메타데이터         |
| `scripts\war3map.j`  | 311,745 bytes | 트리거, 타이머, 자원 루프, 웨이브, 점수 |
| `war3map.w3e`        |  53,040 bytes | 지형 데이터                             |
| `war3map.wpm`        | 118,288 bytes | 경로 데이터                             |

| `war3map.w3a`        | 227,666 bytes | 커스텀 능력 데이터                      |

## 4. 플레이어/팀 구조

JASS `config`와 초기화 구간에서 확인된 구조:

| 항목        | 관찰                           |
| ----------- | ------------------------------ |
| 전체 슬롯   | 12                             |
| 유저 슬롯   | 1~10번 플레이어                |
| 컴퓨터 슬롯 | 11~12번 플레이어               |
| 팀 구조     | 유저는 한 팀, 컴퓨터는 별도 팀 |
| 시작 위치   | 유저 10개 시작 위치 정의       |

Phaser MVP 반영:

- 원본은 최대 10인 구조지만 웹 MVP 내부 테스트는 8인 슬롯을 기준으로 한다.
- 공개 테스트 인원은 성능과 운영 편의에 따라 조절할 수 있으나, 데이터/맵/경제 구조는 "각자 농장 + 공통 늑대 압박"의 원본 근접 감각을 유지한다.

## 6. 초기 지급과 초반 안내

초기화 트리거에서 확인된 흐름:

| 항목         | 값                                 |
| ------------ | ---------------------------------- |
| 초기 골드    | `1500 + 난이도 보정값`             |
| 쉬움 보너스  | 추가 골드 `+200`                   |
| 시작 유닛    | 플레이어별 `n002` 개 1기 + `H000` 농부 1기 |
| 첫 늑대 안내 | "80초 후 부터 늑대들이 출몰"       |

기획 반영:

- 사용자의 "첫 시작때 농부+개" 기억은 JASS 생성 라인과 일치한다. `IIilii` 1736에서 `n002` 개를 먼저 만들고, 1737에서 `H000` 농부를 같은 시작 위치에 생성한 뒤 농부를 선택 유닛으로 지정한다.
- MVP 초기 상태도 플레이어마다 농부 1기와 개 1기를 배치한다. 개는 초반 거미/보너스 사냥과 첫 방어 보조를 담당하는 시작 전투 유닛으로 둔다.
- 웹 MVP에서는 원본 논리 시간 80초 첫 웨이브를 유지하고, 빠른 검증은 `timeScale`로 처리한다.
- 문서의 현재 80초 첫 웨이브 가정은 원본 JASS 안내와 일치한다.

## 7. 닭 경제 루프

JASS 후반부에 30초 주기의 자원 지급 트리거가 있다.

| 항목              |     값 |
| ----------------- | -----: |
| 지급 주기         |   30초 |
| 기본 닭 rawcode   | `h00A` |
| 기본 닭 목재 수입 |     70 |
| 중간 닭 rawcode   | `h00J` |
| 중간 닭 목재 수입 |    110 |
| 고급 닭 rawcode   | `h00W` |
| 고급 닭 목재 수입 |    170 |

원본은 닭 수에 따라 목재가 증가한다. 웹 게임에서는 자원명을 그대로 따르지 않고, 다음 중 하나로 해석한다.

| 원본       | 웹 MVP 후보                       |
| ---------- | --------------------------------- |
| 목재 지급  | 알 판매 수익 또는 농장 생산 수익  |
| 30초 주기  | 알 생산/수확 주기                 |
| 70/110/170 | 닭 등급별 생산량 또는 판매가 배율 |

권장 MVP 변환:

```json
{
    "chickenTiers": [
        { "tier": 1, "sourceRawcode": "h00A", "referenceYield": 70, "mvpEggAmount": 1 },
        {
            "tier": 2,
            "sourceRawcode": "h00J",
            "referenceYield": 110,
            "mvpEggAmount": 2
        },
        { "tier": 3, "sourceRawcode": "h00W", "referenceYield": 170, "mvpEggAmount": 3 }
    ],
    "eggProductionIntervalSec": 30
}
```

### 8.1 늑대 티어/웨이브 보충 구조

늑대 웨이브는 "몇 번째 웨이브에 특정 늑대 N마리"를 직접 생성하는 방식이 아니라, `iIi[1..18]` 티어 배열과 `Ili[1..18]` 목표 생존 수를 조합해 작동한다.

핵심 원리:

- `iIi[1..18]`에는 일반 늑대 rawcode `n007`부터 `n014`까지 18개 티어가 순서대로 들어간다.
- 단계 전환 함수는 특정 티어의 목표 생존 수 `Ili[]`를 즉시 올리고, 기존 보충 트리거를 끄고 다음 보충 트리거를 켠다.

- 따라서 Phaser 구현도 고정 웨이브 큐보다 `phase -> activeTierRange -> targetPopulationDelta -> replenishment` 모델로 옮기는 편이 원본 경험에 가깝다.

### 8.2 늑대 명령 좌표와 대상 우선순위

JASS 늑대 명령 흐름은 `jass_wolf_order_flows.tsv`, `jass_wolf_spawn_points.tsv`, `jass_wolf_order_areas.tsv`로 분리했다.

핵심 결론:

- JASS에는 "수익 건물 -> 닭 -> 플레이어 -> 펜스" 같은 구체적인 대상 우선순위가 직접 하드코딩되어 있지 않다.
- 일반 늑대는 13개 외곽 스폰 rect에서 등장하고, `IlIlI` 전체 공격 구역 안의 랜덤 지점으로 `attack` 명령을 받는다.
- 60초마다 Player(10)의 늑대 전체에 같은 방식의 전역 공격 갱신이 들어간다.
- 실제 공격 대상 선택은 Warcraft III의 attack-move, 자동 target acquisition, pathing, 충돌/막힘 처리에 맡겨진 구조로 보는 편이 맞다.
- 특수 이벤트성 명령은 보스/특수 어그로, 저체력 방어 유닛 집중, 수동 탱커 집중에 가깝고 일반 늑대의 기본 우선순위로 보기는 어렵다.

주요 좌표:

| 영역                  | 원본 변수    | 좌표                              |
| --------------------- | ------------ | --------------------------------- |
| 늑대 전역 공격 구역   | `IlIlI`      | `(-3168, -7424)` ~ `(7008, 2848)` |
| 난이도/특수 생성 구역 | `IliiI`      | 중심 `(2080, -1440)`              |
| 보스 생성 구역        | `IlilI`      | 중심 `(2080, -896)`               |
| 외곽 스폰 지점        | `Iii[1..13]` | `jass_wolf_spawn_points.tsv` 참조 |

확인된 명령 플로우:

| 함수                 |      라인 | 해석                                                                   |
| -------------------- | --------: | ---------------------------------------------------------------------- |
| `IliiIII`            | 5720-5722 | 늑대 스폰 rect에 들어온 Player(10) 유닛을 `IlIlI` 랜덤 지점으로 attack |
| `IliiIlI -> IliiIiI` | 5726-5728 | 60초마다 Player(10) 전체 유닛에 `IlIlI` 랜덤 지점 attack 갱신          |
| `llIIIi -> lillli`   | 3505-3507 | 트리거 유닛 주변 6000 범위 Player(10) 유닛을 해당 위치로 집중          |
| `lIlIIiI -> lIlIIII` | 9419-9430 | 저체력 `n00E` 주변 800 범위 적에게 해당 유닛 공격 명령                 |

Phaser MVP 해석:

- 전략 목표는 "농장 내부 가치 구역"으로 두되, 세부 우선순위는 Warsmash behavior 관찰을 바탕으로 웹 AI 레이어에서 새로 정의한다.
- 펜스/벽은 최우선 공격 대상이 아니라, 경로를 막거나 가까운 blocker가 되었을 때 공격하는 fallback 대상으로 둔다.
- 원본의 60초 전역 공격 갱신은 MVP에서는 idle/stuck 감지 후 3~8초 안에서 목표를 재평가하는 규칙으로 압축한다.

### 10.1 초반 거미와 늑대의 돌 보너스 검증

실제 플레이 기억 중 "초반 거미를 잡아 보너스를 받고, 늑대가 소환되는 돌을 부수면 보너스를 받는다"는 흐름은 정적 분석상 대체로 맞다.

확인 근거:

| 항목 | 정적 분석 근거 | 해석 |
|---|---|---|
| 초반 거미 처치 권장 | `wts_strings.json` 안내문에 "초반 거미를 잡으시면 돈을 주니 꼭 잡으세요" 문구 존재 | 제작자가 의도한 초반 보너스 루프 |
| 거미 배치 | JASS 초기 배치에서 `n01D` 거미 8마리 직접 생성, 좌표는 약 `(6868,-7342)`, `(1581,-7254)`, `(6768,-3275)`, `(1941,2510)`, `(6891,2529)`, `(-2920,2647)`, `(-2776,-7353)`, `(-2802,-2371)` | 8인 테스트 맵의 초반 중립 목표 후보 |
| 거미 처치 드랍 | `n01D` 사망 시 50% 확률로 `I00G` 아이템 생성 | 거미 보너스는 확률 드랍 구조 |
| `I00G` 사용 효과 | 농부 `H000`이 `I00G`를 사용하면 `n011` 중년 닭 생성 | "돈" 안내와 달리 JASS상 확인 가능한 효과는 초반 경제 유닛 보상 |
| 늑대의 돌 | `n00D` 늑대의 돌 13개가 JASS `CreateUnit`으로 직접 배치됨. 좌표는 `key_unit_placement_reference.tsv` 참조 | 외곽 압박/목표 오브젝트 |
| 늑대의 돌 파괴 점수 | `n00D` 사망 시 `IllI += 100` 및 "늑대의 돌을 부셔 100점을 얻으셨습니다" 메시지 | 파괴 보너스는 점수 +100으로 확정 |
| 고대 늑대의 돌 | `n017` 고대 늑대의 돌 1개가 `(2112.0, -704.0)`에 배치됨, HP 150000 | 후반/특수 목표 후보 |

주의할 점:

- "거미를 잡으면 돈"이라는 안내는 WTS에 직접 존재하지만, 현재 추출된 JASS에서 `n01D` 처치 즉시 골드를 지급하는 코드는 확인되지 않았다.
- 확인 가능한 거미 보상은 `I00G` 드랍이며, 이 아이템은 사용 시 `n011` 중년 닭을 만든다. 실제 플레이에서는 이 경제 보상이 돈처럼 체감됐을 가능성이 있다.
- "거미 소환 돌"이라는 표현은 현재 정적 분석만으로는 별도 rawcode를 확정하지 못했다. 확정된 파괴 보너스 오브젝트는 `n00D` 늑대의 돌이다.
- Phaser Tilemap의 늑대의 돌/고대 늑대의 돌/거미/중앙 NPC 좌표는 `phaser_object_position_crosscheck.tsv`에서 원본 JASS 좌표와 교차검증한다.

- `n01J`는 정적 유닛 데이터상 이름이 "동물나라 사신"이며, 문자열에서 "거북이", "구조", "구해달라" 같은 직접 문구는 아직 확인되지 않았다.

- 웹 구현에서는 "중앙 거북이 구조 이벤트"를 원본 감각 후보로 유지하되, 첫 MVP에서는 중앙 허브 이벤트 슬롯과 보상 구조만 먼저 열어두는 편이 안전하다.

현재 작업공간에는 Warcraft III 실행 환경이나 리플레이 파일이 없어 실제 플레이 관찰은 완료하지 못했다. 이 단계에서는 실제 플레이 관찰을 필수 선행 조건으로 두지 않고, `docs/chicken_farm/chicken_farm_warsmash_behavior_notes.md`의 Warsmash behavior 관찰로 워3식 이동/공격/막힘 감각을 갈음한다.

정적 분석상 실제 플레이 또는 후속 도구 검증에서 확인하면 좋은 지점:

## 12. 오브젝트 데이터 추가 분석

`war3map.w3a`, `war3map.w3q`, `war3map.w3d`는 raw object modification 형식으로 파싱 가능했다.

| 파일          | original 엔트리 | custom 엔트리 | 분석 의미         |

| `war3map.w3a` |             314 |           170 | 커스텀 능력/스킬  |
| `war3map.w3q` |              56 |            15 | 커스텀 업그레이드 |

`war3map.w3h`, `war3map.w3b`는 거의 끝까지 파싱되지만 마지막 바이트 정렬 문제로 완전 파싱은 실패했다. 현재 분석에는 `strings`/JASS 보조 근거만 사용한다.

- "알 끌어모으기"는 추후 액티브 스킬 또는 건물 업그레이드로 재해석한다.
- 부화는 건물별 복잡도를 줄이고, MVP에서는 공통 `hatchery` 한 종류로 시작한다.

| -------------------------------------- | ------------------------------------------ | ------------------------ |
| `A001`, `A00Y`                         | 모이주기                                   | 닭 만족도/생산 가속 후보 |
| `A003`, `A00J`, `A02X`                 | 모이 뿌리기, 최대 6/8/16마리 단위          | 범위 버프 또는 닭 유도   |

MVP 반영:

- 첫 버전에서는 모이 시스템을 별도 자원으로 만들지 않고, 닭 생산 가속 버프 정도로 제한한다.
- "닭 보호"는 늑대 웨이브 직전 쓰는 방어 스킬로 좋지만, MVP에서는 우물/울타리/타워가 먼저다.
- 특수 닭은 2차 버전의 전략 다양화 요소로 남긴다.

2026-07-09 Chicken AI 재해석:

- `n000/n011/n01A` 닭 계열은 각각 HP `40/55/65`이며 공통으로 `Awan`을 보유한다.
- `A001/A00Y 모이주기`는 Heal 기반이고 "배고픈 동물"을 대상으로 명시한다.
- 우물 `h00M`은 `A01L 먹이 유혹`과 `A003 모이 뿌리기`를 보유한다.
- 이 조합은 사용자 플레이 기억의 "닭 체력이 감소하고 우물에서 채운다"는 동작과 부합한다. 다만 감소 주기와 회복량은 현재 정적 산출물에서 확정되지 않았다.
- Phaser PoC에서는 닭 HP 자연 감소, 저체력 시 우물 탐색, 우물 범위 회복, 회복 후 `Awan`식 배회를 Chicken AI의 1차 상태 머신으로 구현한다. 상세 단계와 provisional 수치는 `chicken_farm_next_priority_plan.md`의 `Chicken Vitality / Well AI 계획`을 따른다.

### 12.3 아내/가족/테크 트리

사용자 플레이 기억의 "결혼을 통해 아내가 나온다"는 표현은 현재 정적 문자열에서 `결혼`이라는 단어로 직접 확인되지는 않는다. 다만 유닛/건물 교차표에서는 `H002` 아내가 명확히 존재하고, 마을회관/농가/신전 계열에서 가족/보조 유닛을 훈련하는 테크 구조가 확인된다.

핵심 rawcode와 관계:

| rawcode | 이름 | 관찰 근거 | 해석 |
| --- | --- | --- | --- |
| `H000` | 농부 | 시작 시 생성, 건설 목록 보유, 능력 `A01V,A001,A002,Ahrp,A005,A033,AInv` | 플레이어 주 조작 영웅 |
| `H002` | 아내 | `unit_rawcode_crosscheck.tsv`에 독립 유닛으로 존재, 능력 `A04I,A02K,A00U,A00T,A001,A002,AInv` | 결혼/가족 테크의 핵심 보조 캐릭터 후보 |

아내 관련 능력 단서:

| rawcode | 이름/설명 단서 | MVP 해석 |
| --- | --- | --- |
| `A00T` | 치유의 손길, 최대 4기 순환 치료 | 아내/가족의 전투 후 복구 스킬 |

레벨/강화 관련 주의:

- `H000` 농부와 `H002` 아내 모두 unit level 필드는 5로 관찰된다. 이는 에디터 유닛 레벨값이며, 실제 게임 중 경험치 레벨업 구조와 동일하다고 단정하면 안 된다.
- JASS 호출 그래프에는 `AddHeroXPSwapped` 호출 후보가 여럿 있으나, 현재 산출물만으로 농부/아내 중 누구에게 어떤 조건으로 경험치가 들어가는지까지는 좁히지 못했다.
- `A00X`의 툴팁에는 `[레벨 1]`, `A03T`는 1~3레벨, `A03V/A03W`는 `레벨 %d` 배우기 문자열이 있어, 스킬 레벨/학습형 강화가 존재했을 가능성이 높다.
- 웹 MVP에서는 농부/아내/가족 테크를 단순 장식이 아니라 원본 근접 핵심 루프로 취급한다. 단, 세부 레벨업 공식은 추가 JASS 파서 또는 Warsmash 관찰 전까지 후보로 둔다.

MVP 반영:

- `marriage` 또는 `family_unlock` 테크를 두고, 해금 후 `spouse` 캐릭터를 생성/훈련할 수 있게 한다.
- 아내는 전투 캐릭터보다 치료, 닭 몰이, 버프 역할을 우선한다. 질병 치료는 원본 참조 능력으로만 남기고 현재 구현에서는 제외한다.
- 아들/딸/개/큰 개/와이번은 후속 가족/보조 테크 트리로 유지한다.
- PoC 단계에서는 먼저 `spouse` 생성, 치료, 닭 몰이 기능을 좁혀 검증한다.

MVP 반영:

- 배럭은 `mercenary_barracks`로 두고, 방어 유닛을 직접 생산하는 선택지로 유지한다.
- 연구소는 단일 건물이 아니라 `blacksmith -> workshop -> arcane_lab` 계층으로 둔다.
- 타워/개/용병/자동화 연구는 런타임 스탯 보정 테이블로 분리한다.

W3X 정적 추출표의 타워 행은 공격 수치와 모델/업그레이드 라인은 확인되지만, 정확한 건설 pathing texture 크기는 아직 별도 추출 대상이다. 실제 플레이 이미지에서는 타워가 배치 그리드 한 칸을 차지하는 것으로 보이므로, MVP에서는 `tower_* = 1 x 1 major tile = 4 x 4 minor cells`로 둔다. ### 15.4 1x2 방벽과 1칸 틈 기억 검증

사용자 기억의 "1x2 크기 방벽을 1x1 정도 띄워 지어도 늑대는 못 들어오고 농부는 지나갈 수 있었다"는 흐름은 정적 분석만으로 완전 확정할 수는 없지만, Warcraft III pathing/충돌 구조상 충분히 가능한 기억으로 판단한다.

확인/추론 근거:

| 항목 | 근거 | 해석 |
|---|---|---|
| pathing cell 크기 | `war3map.wpm`은 32 world-unit cell 기준 | 웹 PoC의 16px tile x 2 scale = 32px grid와 맞음 |
| 이동 판정 | Warsmash 관찰상 pathfinding은 32 world-unit grid와 유닛 collision size를 함께 사용 | 같은 지형 gap이어도 유닛 크기에 따라 통과 가능 여부가 달라질 수 있음 |
| 농부/늑대 이동 타입 | `H000` 농부와 늑대류 모두 `movetp=foot`이지만, 실제 통과 가능성은 이동 타입만이 아니라 collision/pathing clearance 영향을 받음 | "농부는 통과, 늑대는 막힘"은 엔진 판정상 가능 |
| 방벽/울타리 | `h003` 울타리, `h00L` 청동 울타리, `h00K` 돌 벽 등은 건물형 blocker 후보 | 건설 시 정적 WPM이 아니라 동적 building pathing overlay로 막힘을 만든다고 보는 게 맞음 |
| 1x2 방향 가설 | `h003` 울타리 모델이 `Village_Fenceshort`라 시각적으로 얇고 긴 울타리처럼 보일 수 있음 | 원본 `pathTex=PathTextures\4x4SimpleSolid.tga`가 확인되므로 Phaser PoC의 blocked footprint는 `4 x 4 minor = 1 x 1 major`로 둔다 |
| 현재 W3X 한계 | `war3mapUnits.doo`와 정확한 pathing texture가 아직 미추출 | 방벽 footprint가 정확히 1x2인지, 1칸 gap 판정이 어떤 rawcode에서 발생했는지는 추가 도구 검증 필요 |

Phaser 구현 판단:

- 건설 판정은 32px minor cell을 기본으로 두고, 배치 UX는 4 minor 단위 major grid를 기준으로 둔다.
- 울타리/방벽은 `footprintCells`를 별도 데이터로 둔다. 현재 PoC 기준은 원본 `pathTex=PathTextures\4x4SimpleSolid.tga`에 맞춘 `fence_wood: 4x4 minor = 1x1 major`다.
- 늑대 pathing은 `clearanceCells = 2` 또는 `radiusPx` 기반으로 계산해 1칸 통로를 막힌 것으로 볼 수 있게 한다.
- 농부 pathing은 `clearanceCells = 1`로 두어 같은 1칸 통로를 지나갈 수 있게 한다.
- 이 규칙은 원본 기억을 재현하기 위한 MVP 변환값이며, 실제 워3 pathing texture가 확인되면 조정한다.

### 16.1 라벨별 세부 플로우

아래 플로우는 `jass_function_labels.tsv`, `jass_labeled_call_edges.tsv`, `jass_timer_events.tsv`, `jass_economy_events.tsv`, `jass_create_units.tsv`를 함께 대조한 결과다. 함수명은 난독화되어 있으므로 구현에서는 함수명을 복사하지 말고 역할 단위만 참고한다.

#### 웨이브 플로우

원본 웨이브는 `main`/`config`의 기본 세팅, `iiiIilI`의 트리거 등록, 5600~6800 라인대의 웨이브 단계 함수군으로 나뉜다.

```text
config/main
  -> 플레이어/팀/시작 위치/환경 초기화
  -> iiiIilI: 런타임 트리거와 장기 타이머 등록
  -> IliIilI: 난이도 선택 후 초기 늑대 계열 생성
  -> IliiiiI/IliilII/IliillI/IlilIiI/IliliII: 단계별 특수 웨이브
  -> IlillII~IllllII: 현재 단계 유닛 수가 줄면 Player(10)에 2기씩 보충
  -> iIIIilI~iIlIliI: 웨이브 카운트/상태 증가 계열
```

| 단계        | 근거 함수/라인                                       | 관찰                                                                                    |
| ----------- | ---------------------------------------------------- | --------------------------------------------------------------------------------------- |
| 초기화      | `config` 9685-9706, `main` 9670-9684                 | 시작 위치, 팀, 환경, 주요 초기화 함수 호출                                              |
| 트리거 등록 | `iiiIilI` 7233-7887                                  | `TriggerRegisterTimerEventSingle/Periodic`, 채팅/다이얼로그/유닛 이벤트를 한곳에서 등록 |
| 초기 웨이브 | `IliIilI` 5614-5663                                  | 난이도 선택 이후 Player(10) 소유 늑대 계열 생성과 안내                                  |
| 특수 웨이브 | `IliiiiI` 5741-5763                                  | Blood Wolf 계열로 보이는 단계. 생성, 핑, 시야 공유, 다음 트리거 활성화                  |
| 후속 단계   | `IliilII`, `IliillI`, `IlilIiI`, `IliliII` 5776-5894 | Wild Wolf, Hell Hound, Doom Guard, Archimonde 계열 단계로 추정                          |
| 보충 스폰   | `IlillII`~`IllllII` 5907-6174                        | 현재 단계 유닛 수 조건을 보고 `CreateNUnitsAtLoc(2, iIi[...], Player(10))` 형태로 보충  |
| 점수/상태   | `iiIIliI` 6943-6985                                  | 늑대 관련 안내와 리더보드 값 갱신                                                       |

타이머 등록상 장기 웨이브/이벤트 후보는 60, 120, 230, 300, 600, 1100, 1500, 2000, 2200, 2400, 2600, 2800, 3000초에 분포한다. Phaser MVP에서는 이 전체 길이를 그대로 쓰기보다 12~15분 세션에 맞춰 단계 간격을 압축하는 편이 좋다.

#### 경제 플로우

경제는 교환, 구매, 사망/부활 패널티, 주기 수익으로 분리된다.

```text
상점/유닛 상호작용
  -> lIilii: 골드/목재 교환
  -> lIlIii/lIliii/lIllii: 금화 지불 후 닭 계열 생성
  -> liilIi: 영웅 부활 및 자원 40% 손실
  -> lIiillI: 30초마다 닭 보유 수 기준 목재 지급
```

| 기능        | 근거 함수/라인      | 원본 수치/행동                                                               |
| ----------- | ------------------- | ---------------------------------------------------------------------------- |
| 자원 교환   | `lIilii` 3182-3255  | 100 -> 70, 500 -> 400, 1500 -> 1200, 3000 -> 2400 비율로 골드/목재 상호 교환 |
| 닭 구매 1   | `lIlIii` 3268-3275  | 금화 150 소모, Player(11) 소유 유닛 생성                                     |
| 닭 구매 2   | `lIliii` 3288-3295  | 금화 400 소모, Player(11) 소유 유닛 생성                                     |
| 닭 구매 3   | `lIllii` 3308-3315  | 금화 850 소모, Player(11) 소유 유닛 생성                                     |
| 부활 패널티 | `liilIi` 3415-3445  | 영웅 부활 처리, 골드/목재 현재 보유량의 40% 차감                             |
| 주기 수익   | `lIiillI` 9352-9366 | 30초마다 `h00A`/`h00J`/`h00W` 보유 수에 따라 목재 70/110/170 지급            |

Phaser에서는 교환 UI와 닭 구매 UI를 분리하고, 주기 수익은 서버/호스트 authoritative tick으로 처리하는 것이 좋다. 원본의 부활 패널티 40%는 협동 게임에서 긴장감을 만드는 장치라 MVP에도 유지할 만하다.

#### 질병 플로우

질병은 필수 웨이브라기보다 선택형 압박 모드에 가깝다. 다이얼로그/안내/트리거 활성화 후 일정 시간 생존하는 질병 유닛을 생성한다.

```text
질병 옵션 노출
  -> lliIIi: 질병 다이얼로그 버튼 표시
  -> lliiii: 선택 결과/안내 출력
  -> llliii: 질병 트리거 활성화와 안내
  -> llllIi/llllli: 질병 유닛 생성 후 timed life 적용
```

| 단계      | 근거 함수/라인               | 관찰                                                              |
| --------- | ---------------------------- | ----------------------------------------------------------------- |
| 선택 UI   | `lliIIi` 3547-3556           | `DialogAddButtonBJ`, `DialogDisplayBJ`, `DialogSetMessageBJ` 사용 |
| 안내/분기 | `lliiii` 3575-3592           | 질병 관련 텍스트 출력과 조건부 트리거 실행                        |
| 모드 시작 | `llliii` 3642-3660           | 질병 안내, 트리거 활성화, 시간값 `Ili[60]` 참조                   |
| 효과 생성 | `llllIi`, `llllli` 3667-3686 | Player(10) 또는 중립 적대 유닛 생성, `UnitApplyTimedLife` 적용    |

웹 버전에서는 질병을 별도 룰셋으로 두고, 기본 모드에서는 꺼두는 것이 안전하다. 켠다면 “일정 시간마다 농장 내부에 오염/감염 이벤트 발생”처럼 원본의 timed life 압박만 가져오면 된다.

#### 채팅/명령 플로우

채팅 명령은 두 층이다. `iiiIilI`가 기본 런타임 이벤트를 등록하고, `lIlIiII`가 다수의 채팅 명령 등록 함수를 묶어서 호출한다.

```text
main
  -> lIlIiII: 채팅 명령 등록 묶음
       -> iiiIllI: 동맹 관련 명령
       -> iiiiiII: -전체
       -> iiiiiiI/iiiiilI: -모두
       -> iiiillI/iiilIlI: 조건부 채팅 명령
       -> illlIII: -이름
       -> lIIliiI: 추가 채팅 명령
  -> iiiIilI: 런타임 트리거 등록 허브
```

| 명령군      | 근거 함수/라인                           | 관찰                                               |
| ----------- | ---------------------------------------- | -------------------------------------------------- |
| 등록 허브   | `lIlIiII` 9437-9491                      | 53개 호출을 가진 채팅/동맹 등록 묶음               |
| 전체 채팅   | `iiiiiII` 7929-7942                      | `-전체` 채팅 이벤트 등록                           |
| 모두 채팅   | `iiiiilI` 7947-7960                      | `-모두` 채팅 이벤트 등록                           |
| 동맹 채팅   | `iiiIllI` 7895-7908                      | `-동맹` 관련 채팅 이벤트 등록                      |
| 이름 변경   | `illlIII` 8981-8994, `illillI` 8978-8980 | `-이름` 명령 등록과 `SetPlayerName` 실행           |
| 카메라 조정 | `IIllilI`, `IIlllII` 4273-4285           | 채팅 문자열 파싱 후 `SetCameraFieldForPlayer` 호출 |

Phaser에서는 원본 채팅 명령을 그대로 노출할 필요는 낮다. 개발자용 명령, 이름 변경, 전체/팀 채팅 정도만 현대 UI로 옮기면 충분하다.

#### 동맹 플로우

동맹은 채팅 명령과 다이얼로그가 함께 존재한다. 즉시 전체 동맹을 설정하는 함수와, 다이얼로그에서 개별 플레이어를 고르는 함수군이 분리되어 있다.

```text
채팅/동맹 명령
  -> IiIIilI: 전체 동맹/동맹 상태 계열 처리
  -> IiilliI: 동맹 대상 선택 다이얼로그 구성
  -> IilIIlI~IlIiIlI: 플레이어별 선택 액션
  -> IlIilII: 실제 SetPlayerAllianceStateBJ 적용 helper
```

| 단계            | 근거 함수/라인                | 관찰                                                                             |
| --------------- | ----------------------------- | -------------------------------------------------------------------------------- |
| 전체 동맹       | `IiIIilI` 4323-4350           | `SetPlayerAllianceStateBJ` 호출과 동맹 안내 출력                                 |
| 다이얼로그 구성 | `IiilliI` 4669-4946           | `DialogAddButtonBJ`, `DialogDisplayBJ`, `DialogSetMessageBJ`로 대상 선택 UI 구성 |
| 개별 대상 처리  | `IilIIlI`~`IlIiIlI` 4965-5426 | 버튼 선택 후 `DialogClearBJ`, 안내 출력, 동맹 상태 변경                          |
| 적용 helper     | `IlIilII` 5451-5458           | `SetPlayerAllianceStateBJ` 래퍼성 helper                                         |

웹 MVP에서는 워크래프트식 임의 동맹보다 방 생성 시 팀 고정/협동 고정이 더 단순하다. 단, 원본의 `-동맹`, `-올동맹`, `-동맹상태` 흐름은 P2P 방에서 “팀 상태 보기”와 “협동 초대/수락” UI 참고로 쓸 수 있다.

### 18.2 늑대 이동/공격 감각 재현 포인트

원본에서 확인된 사실:

| 근거                          | 관찰                                                                             |
| ----------------------------- | -------------------------------------------------------------------------------- |
| `war3map.wpm`                 | 기본 지형은 ground open 95.6%로 거의 열린 평면                                   |
| `jass_create_units.tsv`       | 늑대/보스는 대부분 Player(10)에 생성                                             |
| `jass_labeled_call_edges.tsv` | `IssuePointOrderLocBJ`, `IssueImmediateOrderBJ` 계열 명령 존재                   |
| `unit_rawcode_crosscheck.tsv` | 일반 늑대 속도 330, 보스 대부분 속도 330                                         |
| `war3map.doo`                 | 농장 주변 doodad는 장식/경계 감각은 주지만 기본 pathing을 미로처럼 만들지는 않음 |

재현 방향:

1. 늑대는 타워디펜스의 고정 경로 적이 아니라, 열린 지형에서 농장/닭/수익 건물을 향해 들어오는 추적자에 가깝게 둔다.
2. 기본 목표 우선순위는 `수익 건물 -> 닭/방어 유닛 -> 농부/핵심 건물`로 둔다.
3. 펜스가 없으면 거의 직선 추적을 하며, 펜스나 건물로 막히면 가까운 우회로를 찾는다.
4. 우회 실패가 길어지면 워3식으로 근처 blocker를 공격 대상으로 전환한다.
5. 공격은 사거리/쿨다운 기반이며, 타깃이 죽거나 사라지면 즉시 재탐색한다.

### 18.3 펜스/장애물 감각 재현 포인트

원본에 명시적인 “펜스” 이름으로 추출된 유닛 스탯은 아직 별도 확정되지 않았다. 다만 `war3mapImported\GreenGrass_Wall.mdx` 참조 에셋과 doodad/pathing 결과를 보면, 플레이어가 방어선을 직접 만들고 늑대의 접근을 지연시키는 감각이 핵심이다.

Phaser에서의 펜스 모델은 다음을 목표로 한다.

| 속성           | 권장                                 |
| -------------- | ------------------------------------ |
| pathing        | grid cell blocker                    |
| 공격 대상 여부 | 늑대가 목표로 삼을 수 있음           |
| 내구도         | 초반 늑대 3~5회 공격에 파괴되는 수준 |
| 배치 규칙      | 플레이어 농장 구역 내부만 가능       |
| 수리/교체      | MVP에서는 재건설 우선, 수리는 후순위 |
| 시각 감각      | 낮고 연속된 울타리/생울타리 실루엣   |

핵심은 “완벽한 미로 만들기”가 아니라 “잠깐 늑대를 묶어두는 방어선”이다. 원본 pathing이 넓게 열려 있으므로, 웹 버전에서도 펜스는 긴 우회 미로보다 농장 보호용 지연 장치로 두는 편이 원본 감각에 가깝다.

### 18.4 낮/밤, 시야, 미니맵 감각

원본 JASS에는 Warcraft III 엔진의 낮/밤 및 시야 관련 기능을 사용하는 단서가 있다.

| 근거 | 관찰 |
|---|---|
| `jass_labeled_call_edges.tsv` | `SetDayNightModels`, `SetAmbientDaySound`, `SetAmbientNightSound` 호출 확인 |
| `jass_labeled_call_edges.tsv` | `TriggerRegisterGameStateEventTimeOfDay` 호출 확인 |
| `jass_function_labels.tsv` | 초기화 함수 `iiiIilI`와 `main`에서 day/night, ambient sound, terrain fog 계열 호출 확인 |
| `jass_day_night_events.tsv` | `6.00` 시각 이벤트가 `lllllI` 액션으로 연결됨 |
| `jass_night_wolf_stat_candidates.tsv` | 스탯 변경 API 후보 중 time-of-day 액션으로 등록된 함수 없음 |
| `war3map.mmp` | 미니맵 메타데이터 추출 가능 |
| 웨이브 함수 요약 | 일부 특수 웨이브에서 미니맵 핑/시야 공유가 함께 관찰됨 |

해석:

- 원본 맵은 Warcraft III 기본 day/night 렌더링과 ambient sound 설정을 유지하거나 커스텀 초기화한 것으로 보인다.
- `TriggerRegisterGameStateEventTimeOfDay(iiiII,EQUAL,6.00)`는 확인되지만, 연결 액션 `lllllI`는 `PlaySoundBJ`, `lIIiiI(10.00)`, `KillSoundWhenDoneBJ`만 호출한다.
- 정적 JASS 기준으로는 time-of-day 이벤트가 `SetUnitMoveSpeed`, `SetUnitState`, `UnitAddAbilityBJ`, `SetHeroLevelBJ` 같은 스탯 변경 API를 통해 늑대를 강화하는 직접 증거가 없다.
- `jass_night_wolf_stat_candidates.tsv`의 스탯 변경 후보 함수들도 time-of-day 액션으로 등록되지 않았고, 늑대 rawcode 직접 참조도 확인되지 않았다.
- 다만 Warcraft III 엔진 기본 낮/밤 효과는 별도다. 크립/유닛의 낮/밤 시야, 시야 획득 체감, ambient/terrain fog는 플레이어가 “밤에 더 위험하다”고 느끼게 만들 수 있다.
- 플레이 경험상 “밤마다 늑대가 강해진다”는 기억은 실제 스탯 버프라기보다 밤 시야/가시성/웨이브 타이머/사운드 경고가 결합된 체감일 가능성이 있다. 이 부분은 Warsmash 또는 실제 플레이 관찰로 재검증한다.

Phaser 반영 방향:

| 항목 | MVP 처리 |
|---|---|
| 낮/밤 사이클 | `elapsedSec` 기반의 시각 연출 레이어로 구현. 게임 규칙과는 처음에 분리 |
| 밤 연출 | 화면 전체에 약한 청록/남색 overlay, ambient 변화, UI clock 표시 |
| 밤 능력치 | 정적 JASS 직접 증거 없음. MVP 기본값에는 넣지 않고, 선택형 modifier 후보로 보류 |
| FoW | 현재 PoC처럼 플레이어 주변 시야와 탐색 완료 영역을 분리 |
| 미니맵 | 탐색된 영역, 플레이어, 스폰/중립/특수 오브젝트를 축약 표시 |
| 웨이브 연결 | 첫 MVP에서는 timer-driven wave 유지. 이후 밤 전환과 웨이브 경고를 시각적으로 맞춤 |

공격 탐지와 시야:

- W3X 추출표에서 커스텀 타워의 큰 값은 `sight/nsight`가 아니라 `acquire`와 `rangeN1` 쪽에서 확인된다.
- Warsmash 소스 기준 자동 공격 후보는 `acquisitionRange` 안에 있어야 하며, `canBeTargetedBy(...)`에서 현재 source player에게 visible인지도 검사한다.
- 따라서 Chicken Farm 타워는 공격 사거리만으로 어둠 속 적을 때리지 않는다. 타워/건물/유닛 시야가 먼저 FoW current visible을 만들어야 공격 후보가 된다.
- 구현 시 `VisibilitySystem`은 explored와 current visible을 분리해 제공해야 하며, 타워 target acquisition은 current visible만 사용한다.

추가 분석 후보:

1. 완료: `TriggerRegisterGameStateEventTimeOfDay` 연결 액션과 스탯 변경 API 후보를 `jass_day_night_events.tsv`, `jass_night_wolf_stat_candidates.tsv`로 분리했다.
2. Warsmash 또는 실제 플레이 관찰로 밤 시간대의 늑대 이동/획득/공격 체감이 낮과 달라지는지 확인한다.
3. 원본 SLK 기본값까지 교차해 wolf 계열의 `sight/nsight` 기본 적용값을 확정한다.
4. 필요하면 Phaser MVP에 `nightEnemyModifier`를 옵션 필드로만 추가하고 기본값은 `1.0`으로 둔다.

이번 단계에서 `war3map.w3a` / `war3map.w3q` 파싱, JASS 타이머/경제/점수/호출 그래프 산출, 시작 좌표 추출, 시작 유닛 확인, `war3map.doo` 배치 파싱, `war3map.w3e` 지형 파싱, `war3map.wpm` 경로 데이터 파싱, JASS 기능군 라벨링, rawcode별 유닛 스탯 교차확인, 웹 MVP 밸런스 JSON, 웨이브/상점/질병 구현 스펙, 참조용 에셋 추출, 펜스/벽/방어 건물 후보 분리, 배럭/연구소/상위 건물 테크 분리, 연구/업그레이드 제공 건물 분리, 아이템 전체 테이블 분리, JASS 늑대 명령 좌표/흐름 분리는 완료했다.
